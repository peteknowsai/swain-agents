# Part 4: CLAUDE.md + Skills Migration

## Goal

Port the advisor persona, tools config, and skill files from OpenClaw format to Claude Code format. A provisioned Sprite behaves like a real Swain advisor — not just an echo bot.

## What Gets Built

```
sprite/
  templates/
    CLAUDE.md.template         # Main advisor template with {{placeholders}}
  skills/
    onboarding.md              # Migrated from skills/onboarding/
    briefing.md                # Migrated from skills/briefing/
    card-create.md             # Migrated from skills/card-create/
    boat-art.md                # Migrated from skills/boat-art/
    boat-scan.md               # Migrated from skills/boat-scan/
    profile.md                 # Migrated from skills/profile/
    cli.md                     # CLI reference (new — combines TOOLS.md patterns)
  config/
    settings.json.template     # Claude Code settings (MCP servers, permissions)
```

## CLAUDE.md Template

Merges what's currently split across 5 files:
- `AGENTS.md` → system instructions, tool rules
- `TOOLS.md` → CLI reference, tool patterns
- `SOUL.md` (generated by `generateSoul()`) → personality
- `IDENTITY.md` (generated by `generateIdentity()`) → name, style
- `USER.md` (generated by `generateUser()`) → captain profile

Into one `CLAUDE.md` with sections:
1. **Identity** — generated personality (warm dock neighbor, regional expertise)
2. **Captain** — name, userId, phone, boat, marina, experience
3. **The Captain Rule** — banned words, tone guidance (preserved exactly)
4. **Messaging Rules** — `reply()` tool usage, brevity, no plain text output
5. **Tools** — swain CLI path, `--json` flag, skill file locations
6. **Skills Reference** — table of skill files and when to read them
7. **Memory** — where to find/update captain memory

## Skills Migration

Each skill gets reformatted:

**Remove:**
- `message action=send channel=whatsapp target={{phone}}` → `reply("text")`
- `NO_REPLY` / `ANNOUNCE_SKIP` patterns → not needed
- `sessions_spawn(...)` → do the work inline
- OpenClaw frontmatter (triggers, session config) → plain markdown
- WhatsApp-specific references → "iMessage" or just "message"

**Keep:**
- All business logic and decision trees
- CLI commands and argument patterns
- Briefing structure and card logic
- Onboarding flow steps
- Profile update patterns

## Template Rendering

Port the generator functions from `api/templates.ts`:
- `generateSoul()` → Identity section
- `generateIdentity()` → folded into Identity
- `generateUser()` → Captain section

Rendering happens at provision time (Part 2's bootstrap step). Bridge sends captain data to Sprite, channel server's `/bootstrap` endpoint renders the template and writes `/data/CLAUDE.md`.

## Acceptance Criteria

- [ ] CLAUDE.md template renders correctly with test captain data
- [ ] All skills converted — no OpenClaw-isms remain (no `message action=`, no `sessions_spawn`, no `NO_REPLY`)
- [ ] Skills reference real `swain` CLI commands that still work
- [ ] A Sprite with the rendered CLAUDE.md + skills responds in character
- [ ] `reply()` is the only way messages reach the captain — no accidental plain text leaks
- [ ] Claude Code settings.json configures the channel MCP server correctly

## Out of Scope

- Desk agent templates (advisor-only for now — desk agents come later)
- Knowledge DB migration (just the files, not stoolap data)
- Memory migration from VPS (Part 7)

## Estimated Effort

Medium. Most of the work is careful editing — the content exists, it just needs reformatting. Template rendering is straightforward string replacement. The tricky part is making sure nothing breaks in the skill logic during conversion.
